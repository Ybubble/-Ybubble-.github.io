第二章-操作系统概述（一）

2.1操作系统的目标和功能

  • 操作系统是控制应用程序执行的程序，是应用程序和计算机硬件间的接口
2.1.1作为用户/计算机接口的操作系统
  • 须提供一些系统程序以便于开发应用程序，其中一部分称为实用工具或库，它们实现了创建程序、管理文件和控制I/O设备时常用的功能
  • 最重要的系统程序是操作系统
  • 操作系统通常提供的服务：
    • 程序开发
    • 程序运行：一个程序运行时，要把指令和数据加载到内存、初始化I/O设备和文件、准备其他一些资源，操作系统为用户处理这些调度问题
    • I/O设备访问
    • 文件访问控制
    • 系统访问：提供资源和数据的保护、解决资源竞争时的冲突问题
    • 错误检测和响应：
      • 对于每种错误，操作系统必须提供响应以清除错误条件，使其对正在运行的应用的影响最小
      • 响应：终止引起错误的程序、重试操作、简单地给应用程序报告错误
      • 记账：收集对各种资源地利用率地统计数据，监控诸如响应时间之类地性能参数
  • 典型计算机系统中地三种重要接口：
    • 指令系统体系结构（ISA）
      • 定义了机器语言指令系统
      • 硬件与软件的分界线
      • 应用程序和实用程序都可以直接访问ISA，这些程序使用指令系统的一个子集（用户集ISA），操作系统使用操作系统资源的机器语言指令（系统级ISA)
    • 应用程序二进制接口（ABI）
      • 定义了程序间二进制可移植性的标准、操作系统的系统调用接口、在系统中通过ISA能使用的硬件资源服务
    • 应用程序编程接口（API）
      • 允许应用程序访问系统的硬件资源和服务，这些服务由用户级ISA和高级语言库（HLL）调用来提供
      • 让应用软件更容易重新编译并移植到具有相同API的其他系统中
2.1.2作为资源管理的操作系统

  • 操作系统与普通计算机软件的作用相同，即它是由处理器执行的一段程序或一组程序，区别在于程序的意图

  • 操作系统经常会释放控制，而且必须依赖处理器才能恢复控制

  • 操作系统控制处理器使用其他资源，并控制其他程序执行的时机

  • 处理器要去执行其他程序，就必须停止执行操作系统程序。这时操作系统会释放对处理器的控制，处理后用足够长的时间恢复控制权，让处理器准备好下一件工作

  • 操作系统的一部分在内存中，包括**内核程序（kernel或nucleus）**和当前正在使用的其他操作系统程序

  • 内核程序包含操作系统中最常使用的功能

  • 内存的其余部分包含用户程序和数据，它的分配由操作系统和处理器中的存储管理硬件联合控制完成

  • 操作系统在运行一个特定的用户程序时，决定分配给处理器的时间，且在多处理器系统中，这一决定传达到所有处理器

2.2操作系统的发展史

2.2.1串行处理
  • 问题：调度、准备时间
2.2.2简单批处理系统

  • 中心思想：使用称为监控程序（monitor）

  • 作业= =》（按顺序组织）= =》批= =》输入设备= =》（监控程序使用）

  • 每个程序完成处理后返回监控程序，同时监控程序自动加载下一个程序

  • 监控程序角度：

    • 监控程序控制事件的顺序

    • 大部分监控程序必须总是处于内存中并且可以执行，这部分程序称为常驻监控程序（resident monitor），其他部分包括一些实用程序和公用函数作为用户程序的子程序，在需要用到它们的作业开始执行时才被载入

    • 1、监控程序每次从输入设备（通常是卡片阅读机或磁带驱动器）中读取一个作业

      2、当前作业被放置在用户程序区域

      ​ 控制权交给这个作业

      3、作业完成后，控制权返回给监控程序，监控程序立即读取下一个作业

      4、每个作业的结果发送到输出设备

  • 处理器角度：

    • 处理器执行内存中存储的监控程序中的指令，这些指令读入下一个作业并存储到内存中的另一部分

    • 1、处理器执行指令，读入一个作业

      2、处理器遇到监控程序中的分支指令，分支指令指导处理器在用户程序的开始处继续执行

      3、处理器继而执行用户程序中的指令，指导遇到一个结束指令或错误条件

      4、处理器从监控程序中去下一条指令

    • “控制权交给作业”：处理器当前取的和执行的都是用户程序中的指令

      “控制权返回给监控程序”：处理器当前从监控程序中去指令并执行指令

  • 监控程序完成调度功能：一批作业排队等候，处理器尽可能迅速地执行作业，没有任何空闲事件

  • 监控程序改善了作业的准备时间

  • 监控处理程序或批处理操作系统，只是一个简单的计算机程序，它依赖于处理器可从内存的不同部分取指令的能力，交替地获取或释放控制权，并且考虑了硬件功能：

    • 内存保护：

      当用户程序正在运行时，不能改变包括监控程序的内存区域

      or：处理器硬件发生错误→控制权转移给监控程序→取消作业，输出错误信息→加载下一个作业

    • 定时器：防止一个作业独占系统，在每个作业开始时设置

    • 特权指令：

      某些机器指令被设计成特权指令，只能由监控程序执行，处理器在运行一个用户程序时遇到这类指令会发生错误，并将控制权转移给监控程序

      I/O指令属于特权指令，监控程序可以控制所有的I/O设备，还可以避免程序意外地读到下一个作业中地作业控制指令

    • 中断

  • 用户程序以**用户模式（user mode）执行，监控程序以系统态或内核模式（kernel mode）**执行

  • 用户程序和监控程序交替执行

  • 缺点：系统开销

    • 一部分内存交付给监控程序
    • 监控程序消耗了一部分机器时间
2.2.3多道批处理系统

  • 单道程序设计（uniprogramming）

    处理器花费一定的运行时间进行计算，直到遇到一个I/O指令，这时它必须等到I/O指令结束后才能继续进行

  • 多道程序设计（multiprogramming）或多任务处理（multitasking）

    内存空间可以保存操作系统（常驻监控程序）和多个用户程序，执行时进行切换

  • 对多道程序设计最有帮助地硬件是，支持I/O中断和**直接存储器访问（Direct Memory Access，DMA）**的硬件
  • 多道程序设计操作系统比单个程序或单道程序设计（uniprogramming）系统复杂，待运行的多个作业必须保留在内存中，因此需要内存管理（memory management）
  • 使用多道程序设计，可使**批处理（batch processing)**更加有效
2.2.4分时系统

  • 多道程序设计允许处理器同时处理多个批作业，还可以处理多个交互作业。对于后者而言，由于多个用户分享处理器时间，因而该技术称为分时（time sharing）

  • 多个用户可以通过终端同时访问系统，由操作系统控制每个用户程序在很短时间内交替执行

  • 如果有n个用户同时请求服务，若不计操作系统开销，每个用户平均只能得到计算机有效速度的1/n

  • 设计良好的系统相应时间可以接近于专用计算机的响应时间

  • 批处理多道程序 分时
    主要目标 充分利用处理器 减小响应时间
    操作系统指令源 作业控制语言
    作业提供的命令    		 终端键入的命令

  • 时间片（time sharing）技术：在固定时间间隔内，当前用户被抢占，另一个用户被载入

2.3主要成就

2.3.1进程

  • 进程的概念是操作系统设计的核心

  • 定义：

    • 一个正在执行的程序
    • 计算机中正在运行的程序的一个实例
    • 可分配给处理器并由处理器执行的一个实体
    • 由一个单一顺序线程、一个当前状态和一组相关的系统资源所表征的活动单元

  • 计算机系统发展的三条主线

    • 多道程序批处理操作
      • 为了让处理器和I/O设备（包括存储设备）同时保持忙状态，以实现最大效率
      • 关键机制：在响应表示I/O事务结束的信号时，操作系统将对内存中驻留的不同进程进行处理器切换
    • 分时系统
      • 为了能及时响应单个用户的要求
    • 实时事务处理系统
      • 应对场景举例：很多用户都在对数据库进行查询或修改（航空公司订票）
      • 事务处理系统和分时系统的差别：前者局限于一个或几个应用，而后者的用户可以开始程序、执行作业以及使用各种各样的应用程序

  • 程序设计中少见的错误

    • 不确定的同步：常常会出现一个例程必须挂起，等待系统中其他地方的某一事件的情况
    • 失败的互斥：常常会出现多个用户或程序试图同时使用一个共享资源的情况（如同时编辑一个文件）
    • 不确定的程序操作：
      • 某个特定程序的结果只依赖于该程序的输入，而不依赖于共享系统中其他程序的活动
      • 当程序共享内存且处理器控制它们交替执行时，它们可能会因为重写相同的内存区域而发生不可预测的相护干扰
    • 死锁：
      • 很可能两个或多个程序相互挂起等待
      • 如两个程序可能都需要两个I/O设备执行一些操作（如从磁盘复制到磁带）。一个程序获得了一个设备的控制权，而另一个程序获得了另一个设备的控制权，它们都等待对方释放自己想要的资源
      • 这样的死锁依赖于资源分配和释放的时机安排

  • 进程的组成

    • 一段可执行的程序
    • 程序所需要的相关数据（变量、工作时间、缓冲区等）
    • 程序的执行上下文（根本）
      • 执行上下文（execution context）又称为进程状态（process state），是操作系统用来管理和控制进程所需要的内部数据
      • 这种内部信息和进程是分开的，因为操作系统不允许进程直接访问
      • 上下文包括操作系统管理进程及处理器正确执行进程所需的所有信息，包括各种处理器寄存器的内容，如程序计数器和数据寄存器
      • 上下文还包括操作系统使用的信息，如进程优先级和进程是否在等待特定I/O事件的完成

    一种进程管理的办法：

    • 每个进程分配得到了一块存储器区域，并且在操作系统建立和维护的进程表中进行了记录

    • 进程表包含记录每个进程的表项，表项内容包括指向包含进程的存储地址的指针，还包括该进程的部分或全部执行上下文

    • 执行上下文的其余部分存放在别处，可能和进程本身保存在一起，通常还可能保存在内存中的一块独立的区域

    • 进程索引寄存器（process index register）包含当前正在控制处理器的进程在进程表中的索引

    • 程序计数器（program counter）指向该进程的下一条待执行的指令

    • 基址寄存器（base register）和界限寄存器（limit register）定义该进程所占据的存储器区域

      基址寄存器：保存该存储器区域的开始地址

      界限寄存器：保存该区域的大小（以字节或字为单位）

    • 程序计数器和所有数据引用相对于基址寄存器来解释，且不能超过界限寄存器中的值，因此可以保护内部进程不会相互干涉

    • 图中进程索引寄存器表名进程B正在执行，以前执行的进程被临时中断，在A中断的同时，所有寄存器的内容被记录在其执行上下文环境中，以后操作系统就可以执行进程切换，恢复进程A的执行

    • 线程（thread）：一个分配了资源的进程可分解为多个并发的线程，这些线程相互协作执行，完成进程的工作

2.3.2内存管理

  • 操作系统担负着5项存储器管理职责：

    • 进程隔离
      • 操作系统必须保护独立的进程，防止相互干扰各自的存储空间，包括数据和指令
    • 自动分配和管理
      • 程序应该根据需要在存储层次间动态地分配，分配对程序员透明
    • 支持模块化程序设计
    • 保护和访问控制
    • 长期存储

  • （典型）操作系统使用虚存和文件系统机制来履行这些职责

  • 对操作系统来说，文件是访问控制和保护的一个有用单元

  • 虚存机制允许程序以逻辑方式访问存储器，而不考虑物理内存上可用的空间数量

    • 满足多个用户作业同时驻留在内存中的要求，一个进程被写出到辅助存储器中且后续进程被读入时，连续的进程执行之间不会脱节

  • 进程大小不同时，若处理器在很多进程间切换，则很难将它们紧密的压入内存==》引入分页系统

    • 进程由许多固定大小的块组成，这些块被称为页

    • 程序通过虚地址（virtual address）访问字，虚地址=页号+页中的偏移量

    • 分页系统提供了进程中使用的虚地址和内存中的实地址（real address）或物理地址之间的动态映射

    • 虚存（virtual memory）

      • 一个进程的所有页都保存在磁盘中，进程执行时，一部分页会调入内存中

      • 若要访问的某页不在内存中，存储管理硬件会在检测到它后，安排载入这个缺页（missing page）

  • 虚拟处理器

    • 处理器硬件和操作系统共同提供”虚拟处理器“的概念，有对虚存的访问权

    • 给每个进程唯一的不重叠的虚存空间==》实现进程隔离

      两个虚存空间一部分重叠==》内存共享

    • 文件或其一部分可以复制到虚存中提供程序操作

      • 存储器由内存和低速辅助存储器组成，内存可以直接访问（通过机器指令），外存可以通过把块载入内存间接访问
      • 如果访问的虚地址不在实际内存中，实际内存中的一部分内容将换到外存中，然后换入所需要的数据块
2.3.3信息保护和安全

与操作系统相关的大多数安全和保护问题可以分为4类

  • 可用性：保护系统不被中断
  • 保密性：保证用户不能读取未授权访问的数据
  • 数据完整性：保护数据不被未授权修改
  • 认证：涉及用户身份的正确认证和消息或数据的合法性
